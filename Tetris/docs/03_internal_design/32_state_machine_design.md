## 1. 本書の目的

本書は、本プロジェクトにおける**ランタイム状態管理の設計**を定義するための文書である。  
本書では、ゲーム全体がどのような状態を持ち、どの条件で状態が遷移し、各状態で何を受け付け、何を停止し、何を実行するかを整理する。

本書が扱う主題は以下である。

- 起動から終了までに存在する主要状態
- 状態ごとの責務
- 状態ごとの有効入力 / 無効入力
- 遷移条件
- プレイ中状態の内部サブ状態
- 状態管理上の設計原則

本書では以下は詳細定義しない。

- UI の見た目そのもの
- ピース回転テーブルの具体値
- 得点式やレベル式の詳細数値
- 実装言語ごとのクラス設計
- テストケース詳細

---

## 2. 設計方針

本プロジェクトの状態遷移設計は、**Game Boy版相当の単純で追いやすいゲーム進行**を土台としつつ、DocDD見本として曖昧さを減らすことを目的とする。

設計方針は以下の通りとする。

1. 状態は役割ごとに明確に分ける
2. 画面状態とゲーム進行状態をできるだけ対応づける
3. プレイ中だけは内部サブ状態を持ち、ルール処理順序を明確化する
4. 一時停止やゲームオーバー時に、通常プレイ入力が流入しないようにする
5. 遷移条件は外部仕様と矛盾しないよう定義する
6. 実装都合による隠れ状態を増やしすぎない
7. 同じ入力列に対して同じ遷移結果になりやすい設計を優先する

---

## 3. 状態設計の考え方

本プロジェクトでは、状態を大きく以下の2層で扱う。

### 3.1 上位状態

プレイヤーから見て認識可能な大きな状態を指す。

- 起動準備
- タイトル
- 開始設定
- プレイ中
- 一時停止
- ゲームオーバー / リザルト

### 3.2 下位状態

主にプレイ中において、1フレームまたは1処理単位の中で何を行うかを整理するための内部状態を指す。

- ピース出現
- 入力 / 落下処理
- 接地 / 固定判定
- ライン消去処理
- 次ピース準備
- 終了判定

本書では、上位状態を主軸としつつ、プレイ中のみ下位状態も定義する。

---

## 4. 上位状態一覧

本プロジェクトの主要上位状態を以下に示す。

| 状態ID | 状態名 | 位置付け |
|---|---|---|
| ST-BOOT | 起動準備状態 | 内部状態 |
| ST-TITLE | タイトル状態 | 必須 |
| ST-SETUP-A | A-TYPE 開始設定状態 | 必須 |
| ST-PLAY | プレイ中状態 | 必須 |
| ST-PAUSE | 一時停止状態 | 必須 |
| ST-GAMEOVER | ゲームオーバー / リザルト状態 | 必須 |
| ST-MODE-SELECT | モード選択状態 | 予約 |
| ST-SETUP-B | B-TYPE 開始設定状態 | 予約 |

補足:
- 初期実装では A-TYPE を主対象とする
- B-TYPE 関連状態は将来拡張用の予約として扱う

---

## 5. 上位状態遷移概要

基本遷移は以下の通りとする。

```text
ST-BOOT
  ↓
ST-TITLE
  ↓
ST-SETUP-A
  ↓
ST-PLAY
  ├─ START入力 → ST-PAUSE
  └─ ゲームオーバー条件成立 → ST-GAMEOVER

ST-PAUSE
  ├─ 再開入力 → ST-PLAY
  └─ タイトル復帰入力 → ST-TITLE

ST-GAMEOVER
  ├─ 再試行 → ST-SETUP-A
  └─ タイトルへ戻る → ST-TITLE
```

将来拡張時は以下を追加してよい。

```text
ST-TITLE
  ↓
ST-MODE-SELECT
  ├─ A-TYPE選択 → ST-SETUP-A
  └─ B-TYPE選択 → ST-SETUP-B

ST-SETUP-B
  ↓
ST-PLAY
```

## 6. ST-BOOT 起動準備状態

### 6.1 目的

アプリケーション起動直後に、最低限必要な初期化を行う状態である。

### 6.2 主な責務

- 基本設定値の読み込みまたは既定値の適用
- 必要なランタイム資源の初期化
- 画面系初期化
- 入力系初期化
- タイトル画面へ遷移可能な状態の準備

### 6.3 有効入力

原則として通常プレイヤー入力は受け付けない。

### 6.4 遷移条件

初期化完了後、ST-TITLE へ遷移する。

### 6.5 異常時方針

設定未読込や保存未整備などの軽微な問題があっても、既定値で起動継続できる設計を優先する。

---

## 7. ST-TITLE タイトル状態

### 7.1 目的

ゲーム開始前の入口として機能する状態である。

### 7.2 主な責務

- タイトル画面表示
- 開始入力の受付
- 必要最小限の案内表示

### 7.3 有効入力

- ゲーム開始入力
- 将来拡張時はモード選択への入力

### 7.4 無効入力

- プレイ中専用入力（左右移動、回転、ソフトドロップ、一時停止）

### 7.5 遷移条件

- 開始入力 → ST-SETUP-A
- 将来拡張採用時、モード選択入力 → ST-MODE-SELECT

---

## 8. ST-SETUP-A A-TYPE 開始設定状態

### 8.1 目的

A-TYPE 開始前に、開始レベル等の初期条件を設定する状態である。

### 8.2 主な責務

- 設定画面表示
- 開始レベルの変更受付
- 開始確定受付
- 戻る操作受付

### 8.3 有効入力

- 項目変更入力
- 決定入力
- 戻る入力

### 8.4 無効入力

- プレイ中専用入力
- 一時停止入力

### 8.5 遷移条件

- 開始確定 → ゲームセッション初期化後 ST-PLAY
- 戻る入力 → ST-TITLE

### 8.6 遷移時処理

開始確定時には、少なくとも以下を行う。

- 盤面初期化
- スコア初期化
- ライン数初期化
- レベル初期化
- 次ピース生成準備
- プレイ中サブ状態の初期化

---

## 9. ST-PLAY プレイ中状態

### 9.1 目的

実際のゲーム進行を行う主状態である。

### 9.2 主な責務

- ピース出現
- 入力受付
- 時間経過による落下
- 接地 / 固定判定
- ライン消去
- 得点 / ライン / レベル更新
- ゲームオーバー判定
- 次ピース反映

### 9.3 有効入力

- 左移動
- 右移動
- 回転
- ソフトドロップ
- 一時停止

### 9.4 無効入力

- タイトル画面専用入力
- 設定画面専用入力
- リザルト画面専用入力

### 9.5 上位遷移条件

- START入力 → ST-PAUSE
- ゲームオーバー条件成立 → ST-GAMEOVER

### 9.6 下位状態

ST-PLAY 内では、処理順序を明確化するため、以下のサブ状態を持つ。

- PST-SPAWN
- PST-ACTIVE
- PST-LOCK
- PST-CLEAR
- PST-POST-CLEAR
- PST-CHECK-END

詳細は後節で定義する。

## 10. ST-PAUSE 一時停止状態

### 10.1 目的

プレイ進行を中断し、再開または離脱を可能にする状態である。

### 10.2 主な責務

- 一時停止表示
- 再開入力受付
- 必要に応じてタイトル復帰入力受付

### 10.3 有効入力

- 再開入力
- タイトルへ戻る入力（採用する場合）

### 10.4 無効入力

- 左右移動
- 回転
- ソフトドロップ
- その他の通常プレイ入力

### 10.5 停止対象

ST-PAUSE 中は、少なくとも以下を停止しなければならない。

- 自動落下
- 接地 / 固定判定進行
- ライン消去進行
- 得点更新進行
- 通常プレイ入力処理

### 10.6 遷移条件

- 再開入力 → ST-PLAY
- タイトル復帰入力 → ST-TITLE

### 10.7 再開時方針

再開時は、一時停止直前のプレイ中サブ状態へ復帰する。  
ただし、実装上は ST-PLAY 内の安全な再開点へ正規化してもよい。

---

## 11. ST-GAMEOVER ゲームオーバー / リザルト状態

### 11.1 目的

ゲーム終了をプレイヤーへ提示し、再試行またはタイトル復帰へ導く状態である。

### 11.2 主な責務

- ゲームオーバー表示
- 最終結果表示
- 再試行入力受付
- タイトル復帰入力受付

### 11.3 有効入力

- 再試行
- タイトルへ戻る

### 11.4 無効入力

- 左右移動
- 回転
- ソフトドロップ
- 一時停止

### 11.5 遷移条件

- 再試行入力 → ST-SETUP-A
- タイトル復帰入力 → ST-TITLE

### 11.6 保持情報

この状態では、少なくとも以下を参照可能でなければならない。

- 最終スコア
- 最終ライン数
- 最終レベル
- 必要に応じて開始レベル

### 12.1 ST-MODE-SELECT

将来、A-TYPE / B-TYPE のモード選択を導入する場合の予約状態とする。

### 12.2 ST-SETUP-B

## 13. ST-PLAY 内サブ状態一覧

プレイ中は単一状態に見えるが、内部では処理責務が異なるため、以下のサブ状態へ分割する。

| サブ状態ID | 名称 | 主目的 |
|---|---|---|
| PST-SPAWN | 出現処理状態 | 新しい現在ピースの出現 |
| PST-ACTIVE | 操作・落下状態 | 通常操作と自動落下 |
| PST-LOCK | 固定処理状態 | 接地済みピースの固定 |
| PST-CLEAR | ライン消去判定 / 消去状態 | 完成ライン処理 |
| PST-POST-CLEAR | 消去後更新状態 | 得点・ライン・レベル更新 |
| PST-CHECK-END | 終了確認状態 | ゲーム継続可否確認 |

---

## 14. PST-SPAWN 出現処理状態

### 14.1 目的

次ピースを現在ピースとして出現させる。

### 14.2 主な責務

- 次ピースを現在ピースへ反映
- 新しい次ピースを準備
- 初期向き / 初期位置を適用
- 出現可否判定

### 14.3 有効入力

原則として通常プレイ入力はまだ反映しない設計を推奨する。  
ただし、実装簡略化のため、同フレーム中に PST-ACTIVE へ遷移後入力処理してもよい。

### 14.4 遷移条件

- 出現成功 → PST-ACTIVE
- 出現失敗 → PST-CHECK-END

---

## 15. PST-ACTIVE 操作・落下状態

### 15.1 目的

プレイヤー操作と時間経過に基づき、現在ピースを動かす。

### 15.2 主な責務

- 左右移動入力処理
- 回転入力処理
- ソフトドロップ入力処理
- 自動落下処理
- 下方向移動可否判定
- 接地判定

### 15.3 有効入力

- 左移動
- 右移動
- 回転
- ソフトドロップ
- 一時停止

### 15.4 遷移条件

- START入力 → ST-PAUSE
- 下方向移動不能 → PST-LOCK
- それ以外 → PST-ACTIVE 継続

### 15.5 設計方針

本サブ状態では、有効位置判定に基づく入力処理を一貫して行う。  
無効位置となる入力は不成立とし、位置や向きを維持する。

---

## 16. PST-LOCK 固定処理状態

### 16.1 目的

接地済み現在ピースを盤面へ固定する。

### 16.2 主な責務

- 現在ピースの占有マスを固定済みブロックへ変換
- 現在ピースとしての操作対象を終了
- ライン完成判定準備

### 16.3 有効入力

通常プレイ入力は受け付けない。

### 16.4 遷移条件

- 固定完了 → PST-CLEAR

### 16.5 設計方針

本プロジェクトでは Game Boy版相当の単純さを優先し、下方向移動不能となったピースは固定対象として扱う。  
近代的な長い lock delay は設けない。

---

## 17. PST-CLEAR ライン消去判定 / 消去状態

### 17.1 目的

固定後に完成ラインを判定し、必要に応じて消去する。

### 17.2 主な責務

- 完成ライン検出
- 消去対象ライン確定
- 消去実行
- 上部ブロック詰め処理

### 17.3 有効入力

通常プレイ入力は受け付けない。

### 17.4 遷移条件

- 消去対象あり → PST-POST-CLEAR
- 消去対象なし → PST-CHECK-END

### 17.5 設計方針

同時完成した複数ラインは、同一処理単位としてまとめて扱う。

---

## 18. PST-POST-CLEAR 消去後更新状態

### 18.1 目的

ライン消去結果に基づき、進行情報を更新する。

### 18.2 主な責務

- 得点加算
- ライン数更新
- レベル更新
- 落下速度更新

### 18.3 有効入力

通常プレイ入力は受け付けない。

### 18.4 遷移条件

- 更新完了 → PST-CHECK-END

### 18.5 設計方針

得点、ライン、レベル更新は、固定処理や出現処理から分けて扱うことで責務を明確化する。

---

## 19. PST-CHECK-END 終了確認状態

### 19.1 目的

ゲーム継続可否を確認し、次のプレイサイクルへ進めるかを判断する。

### 19.2 主な責務

- ゲームオーバー条件確認
- 次ピース出現可否確認
- 継続時の次サイクル接続

### 19.3 有効入力

通常プレイ入力は受け付けない。

### 19.4 遷移条件

- 継続可能 → PST-SPAWN
- 継続不能 → ST-GAMEOVER

### 19.5 設計方針

終了条件判定を 1 箇所へ集約することで、出現失敗や継続不能条件の追跡を容易にする。

## 20. プレイ中サブ状態遷移概要

ST-PLAY 内の基本遷移は以下の通りとする。

```text
PST-SPAWN
  ├─ 出現成功 → PST-ACTIVE
  └─ 出現失敗 → PST-CHECK-END

PST-ACTIVE
  ├─ START入力 → ST-PAUSE
  ├─ 下方向移動可能 → PST-ACTIVE 継続
  └─ 下方向移動不能 → PST-LOCK

PST-LOCK
  ↓
PST-CLEAR
  ├─ 消去あり → PST-POST-CLEAR
  └─ 消去なし → PST-CHECK-END

PST-POST-CLEAR
  ↓
PST-CHECK-END
  ├─ 継続可能 → PST-SPAWN
  └─ 継続不能 → ST-GAMEOVER
```

## 21. 入力処理方針

### 21.1 上位状態ごとの入力制御

入力は、現在の上位状態および必要に応じてサブ状態に応じて有効 / 無効を判定しなければならない。

### 21.2 無効入力の扱い

無効な状態で受け取った入力は、原則として無視する。  
エラー扱いにはしなくてよい。

### 21.3 プレイ中入力の対象

プレイ中の通常入力は、原則として PST-ACTIVE のみで意味を持つ。  
PST-LOCK, PST-CLEAR, PST-POST-CLEAR, PST-CHECK-END では通常プレイ入力を適用してはならない。

### 21.4 同時入力

同時入力や近接入力の優先順位は、本書では厳密固定しない。  
ただし、以下を満たすこと。

- 同一条件で結果がぶれにくいこと
- 不正位置を生成しないこと
- テスト可能であること

必要に応じて後続文書または ADR で優先順位を固定する。

## 22. 一時停止設計方針

### 22.1 一時停止可能な範囲

一時停止は、原則として ST-PLAY 中にのみ受け付ける。

### 22.2 一時停止復帰

一時停止からの復帰後は、停止前のプレイ文脈が継続されるべきである。

### 22.3 安全再開点

実装上の簡潔さを優先する場合、再開先を以下のように正規化してよい。

- 停止前が PST-ACTIVE の場合 → PST-ACTIVE へ復帰
- 停止前がその他サブ状態であった場合 → そのサブ状態継続、または安全な次サブ状態へ復帰

ただし、外部から見て不自然な結果になってはならない。

## 24. 状態遷移の禁止事項

以下は本プロジェクトで避けなければならない。

### 24.1 不明確な直接遷移

- ST-TITLE からいきなり ST-PLAY へ飛ぶ（開始設定画面を省略する設計を採用した場合は、その理由を記録すること）
- ST-GAMEOVER 中に通常プレイ状態へ自然復帰する
- ST-PAUSE 中にプレイ入力で盤面が進行する

### 24.2 サブ状態飛び越しによる責務混在

- 固定とライン消去と得点更新を区別なく同時処理して設計上の責務を不明瞭にする
- 終了判定位置が複数箇所に散る

### 24.3 状態と画面の不整合

- タイトル表示中なのに内部状態は ST-PLAY
- 一時停止表示中なのに落下が進行する
- ゲームオーバー表示中なのに回転入力が受理される

## 25. 異常系設計方針

### 25.1 想定内異常

以下のようなものは想定内異常として扱い、致命的破綻を避ける。

- 無効状態での入力
- 不要タイミングでの決定入力
- 連打による重複入力

### 25.2 想定外不整合

以下のようなものは、検出しやすい形で扱うことが望ましい。

- ST-PLAY なのに active_piece が存在しない
- PST-ACTIVE なのに board が未初期化
- ST-GAMEOVER なのに game_over_flag が偽

### 25.3 方針

想定外不整合を黙って継続するよりも、ログやデバッグ検出しやすい構造を推奨する。

## 27. 受入観点（本書レベル）

本書に基づく状態遷移設計の受入観点は、少なくとも以下とする。

- 上位状態が定義されていること
- プレイ中サブ状態が定義されていること
- 各状態の責務が明確であること
- 有効入力 / 無効入力が状態ごとに整理されていること
- 一時停止中に通常プレイ進行が止まることが設計上明確であること
- ゲームオーバー遷移位置が追跡できること
- 固定、消去、更新、終了確認の責務分離が成立していること

---

## 28. 結論

本プロジェクトの状態遷移設計は、上位状態で画面・大局進行を管理し、プレイ中のみサブ状態でルール処理順序を管理する構成とする。

中核となる考え方は以下である。

- タイトル、設定、プレイ、一時停止、ゲームオーバーを明確に分ける
- プレイ中は出現、操作、固定、消去、更新、終了確認を分ける
- 有効入力を状態ごとに制御する
- 一時停止やゲームオーバーで通常プレイ処理を止める
- ゲーム進行の責務境界を明確にする

この設計により、DocDD見本として追いやすく、実装時にも混乱しにくいランタイム状態管理を成立させる。
