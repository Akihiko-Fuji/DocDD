## 1. 本書の目的

本書は、本プロジェクトにおける**品質確認の考え方と、テストをどのように設計・追跡・運用するか**を定義するための文書である。

本プロジェクトにおけるテストは、単に「動くかどうか」を確認する活動ではない。  
DocDD（Document-Driven Development）の考え方に基づき、**要件定義からテストまでを一貫して追跡可能にし、「なぜこの実装があるのか」を常に辿れる状態を維持すること**を目的とする。

本書では、以下を定義する。

- テストの基本思想
- DocDD における追跡軸
- 本プロジェクトにおける試験観点
- テストレベル
- 文書とテストの対応付け
- 変更時の影響追跡方法
- 完了判定の基本方針

一方で、本書では以下は詳細定義しない。

- 個別テストケースの全件一覧
- 実際のテスト手順書
- 実装コードに依存したテストコード詳細
- 個別バグ票の運用詳細

それらは後続文書へ委譲する。

## 2. DocDD におけるテストの位置付け

本プロジェクトでは、テストを開発末尾の確認作業としてではなく、**上位文書の妥当性を下位成果物で検証するための連結点**として扱う。

テストは、少なくとも以下の役割を持つ。

1. 要求が仕様へ正しく落ちているかを確認する
2. 仕様が設計へ正しく反映されているかを確認する
3. 設計どおりに実装されているかを確認する
4. 実装により要求を満たしているかを確認する
5. 変更時に影響範囲を可視化する
6. 「意図した仕様」なのか「偶然動いているだけ」なのかを区別する

したがって、本プロジェクトでは、**テストケースは要求・仕様・設計から派生しなければならない**。  
コードを書いた後に思いつきでテストを足すだけでは、DocDD の見本としては不十分である。

## 3. 7つの追跡軸と BR の位置付け

本プロジェクトでは、DocDD の基本思想として、**BR を起点とし、7つの主要軸を経由して TC まで追跡する**考え方を採用する。

### 3.1 基本構造

```text
BR → UC → DM → SR / NSR → EXT → API → TC
```

### 3.2 各略語の意味

| 略語 | 正式名称 | 説明 |
|---|---|---|
| BR | Business Requirement | ビジネス要求。なぜそのプロジェクトを行うのか、何を達成したいのかを定義する上流要求 |
| UC | Use Case | 利用者視点の利用場面。誰が、何の目的で、どう使うかを表す |
| DM | Domain Model | ドメインモデル。対象世界の概念、状態、関係、責務を整理する |
| SR | System Requirement | 機能要件。システムが何を提供すべきかを定義する |
| NSR | Non-System Requirement / Non-Functional Requirement | 非機能要件。品質、応答性、保守性、再現性などを定義する |
| EXT | External Interface / External Specification | 外部仕様・外部接点。プレイヤーや外部入出力から見える振る舞いを定義する |
| API | Public Contract / Callable Interface | 実装や試験の対象となる公開インターフェース。HTTP API に限らず、モジュール外から利用・検証される呼び出し契約全般を指す |
| TC | Test Case | テストケース。上位文書の内容を確認する検証項目 |

### 3.3 「7つの軸」の数え方

本書では、BR を上流起点として扱い、その下流にある以下の7つを主要追跡軸とみなす。

1. UC
2. DM
3. SR
4. NSR
5. EXT
6. API
7. TC

つまり、BR は起点であり、7つの主要軸は BR 以降の追跡対象群である。

## 5. テスト戦略の基本原則

本プロジェクトにおけるテストは、以下の原則に従う。

### 5.1 文書起点

テストケースは、コードから逆算して作るのではなく、要求・仕様・設計から導出する。

### 5.2 追跡可能性

各テストケースは、可能な限り以下を辿れる状態でなければならない。

- どの BR / UC / DM / SR / NSR / EXT / API に由来するか
- 何を確認するためのテストか
- 仕様変更時にどのテストが影響を受けるか

### 5.3 役割分離

テスト方針、テストケース、性能確認、完了条件は文書を分けて管理する。

### 5.4 コア優先

初期段階では、ゲームとして重要なコアロジックと状態遷移を優先して試験対象とする。  
装飾や周辺機能は後順位とする。

### 5.5 境界条件重視

落下ブロックゲームは、壁際、床際、積み上がり、出現不能、同時消去など、境界条件で不具合が出やすい。  
そのため正常系のみで完了扱いにしない。

### 5.6 無効入力も対象

「押してはいけない状態で押したらどうなるか」も試験対象とする。  
特に一時停止中、ゲームオーバー後、タイトル画面などでは重要である。

### 5.7 小規模でも実務粒度

見本プロジェクトであるため、テストを過剰に巨大化させない。  
ただし、「小規模だから雰囲気で済ませる」ことは避ける。

## 8. テストレベル

本プロジェクトでは、テストを以下のレベルで整理する。

### 8.1 L1: 文書整合レビュー

**目的:** 文書間の矛盾を検出する。要求から仕様、設計、テストのつながりを確認する。

対象例:
- `13_functional_requirements.md` と `20_game_rules_spec.md` の整合
- `21_ui_screen_spec.md` と `32_state_machine_design.md` の整合
- `24_piece_rotation_collision_spec.md` と `43_test_cases_edge_conditions.md` の整合

### 8.2 L2: ルール単位試験

**目的:** 個別ルールが仕様どおりに成立するか確認する。

対象例:
- 1ライン消去
- 4ライン同時消去
- ソフトドロップ
- 出現不能時のゲームオーバー

### 8.3 L3: 状態遷移試験

**目的:** 上位状態およびプレイ中サブ状態の遷移が設計どおりか確認する。

対象例:
- タイトル → 設定 → プレイ
- プレイ → 一時停止 → 再開
- プレイ → ゲームオーバー
- PST-ACTIVE → PST-LOCK → PST-CLEAR → PST-POST-CLEAR → PST-CHECK-END

### 8.4 L4: 画面・入力試験

**目的:** プレイヤー視点で画面と入力の組み合わせが正しく動くか確認する。

対象例:
- タイトル画面で開始入力のみ有効
- 一時停止中に落下操作が無効
- ゲームオーバー画面で再試行ができる

### 8.5 L5: 境界条件試験

**目的:** 不具合が起こりやすい端条件で破綻しないか確認する。

対象例:
- 左壁密着状態での回転失敗
- 床際回転失敗
- 固定済みブロックとの干渉
- 高速入力時の不正位置発生防止

### 8.6 L6: 非機能確認

**目的:** 品質要求を満たすか確認する。

対象例:
- 入力に対する違和感のない応答
- 同一入力列に対する同一結果
- 長時間プレイ時の重大破綻なし
- 文書とテストケースの追跡可能性

## 9. 試験観点の分類

本プロジェクトでは、テスト観点を以下の分類で管理する。

### 9.1 ルール観点

- ゲームルールどおりに動くか
- 得点 / ライン / レベルが正しいか
- ゲーム終了条件が正しいか

### 9.2 入力観点

- 有効入力が機能するか
- 無効入力が無害に無視されるか
- 状態依存で入力解釈がぶれないか

### 9.3 状態観点

- 状態遷移が正しいか
- 一時停止中に進行しないか
- ゲームオーバー後に通常プレイへ戻らないか

### 9.4 画面観点

- 必要情報が表示されるか
- プレイヤーが現在状態を認識できるか
- UI が盤面視認性を阻害しないか

### 9.5 境界観点

- 壁際 / 床際 / 積み上がり際で正しいか
- 出現不能条件が正しく扱われるか
- 同時消去が正しいか

### 9.6 品質観点

- 応答性
- 一貫性
- 再現性
- 保守しやすさ
- テストしやすさ

## 10. 軸ごとのテスト観点

本節では、7つの追跡軸ごとに、テストで何を確認するかを整理する。

### 10.1 BR 起点確認

BR は直接テストケース化されるものではなく、何のためにその機能・仕様・設計が存在するかの確認起点として使う。

確認例:
- 本プロジェクトが「DocDD 見本」であることに照らし、文書とテストがつながっているか
- ゲーム完成だけでなく、追跡可能性が成果物になっているか

### 10.2 UC 観点

ユースケースごとに、プレイヤーが目的を達成できるかを確認する。

確認例:
- プレイヤーが新規ゲームを開始できる
- プレイヤーがゲーム中に一時停止できる
- プレイヤーがゲーム終了後に再試行できる

### 10.3 DM 観点

ドメイン概念や状態遷移が破綻していないかを確認する。

確認例:
- `active_piece` が存在しないのに PST-ACTIVE へ入らない
- `game_over` 状態で通常プレイ入力が効かない
- `board` と表示情報が矛盾しない

### 10.4 SR 観点

機能要求を満たしているかを確認する。

確認例:
- FR-101 ピース出現
- FR-204 ライン消去
- FR-401 一時停止
- FR-405 状態遷移管理

### 10.5 NSR 観点

非機能要求を満たしているかを確認する。

確認例:
- NFR-001 基本操作応答性
- NFR-103 プレイ結果の再現可能性
- NFR-202 責務分離
- NFR-502 文書間追跡可能性

### 10.6 EXT 観点

プレイヤーから見える外部仕様どおりに振る舞うか確認する。

確認例:
- `20_game_rules_spec.md` の得点表どおりか
- `21_ui_screen_spec.md` の画面遷移どおりか
- `24_piece_rotation_collision_spec.md` の wall kick 非採用が守られているか

### 10.7 API 観点

ゲームコアの公開境界に対して、期待される契約が守られるか確認する。

確認例:
- 新規ゲーム開始操作が初期状態を正しく生成する
- 入力受付が状態依存で正しく制御される
- 1 tick / 1 step の進行で不正状態を作らない

### 10.8 TC 観点

テストケース自体が上流文書を参照可能であり、何を確認しているか明確であることを確認する。

## 11. トレーサビリティ管理方針

### 11.1 基本方針

本プロジェクトでは、テストケースを単独で管理しない。  
各テストケースは、少なくとも以下のいずれかと結び付いていなければならない。

- FR
- NFR
- EXT
- 状態設計項目
- 境界条件仕様

### 11.2 推奨トレース単位

最低限、以下の対応を持つことを推奨する。

| 上流 | 下流 |
|---|---|
| FR | TC |
| NFR | TC |
| EXT | TC |
| State | TC |

### 11.3 理想的トレース

可能であれば、以下を辿れる状態とする。

| 起点 | 追跡先 |
|---|---|
| BR | UC |
| UC | DM |
| DM | SR / NSR |
| SR / NSR | EXT |
| EXT | API |
| API | TC |

### 11.4 変更時の影響追跡

仕様変更時は、次の観点で影響を追う。

- 変更元はどの文書か
- どの FR / NFR に影響するか
- どの EXT / State / API に影響するか
- どの TC を更新すべきか
- どの Change Log / Decision Log を更新すべきか

## 12. テストケース識別子方針

本プロジェクトでは、テストケースはカテゴリごとに ID を持たせることを推奨する。

### 12.1 推奨形式

- `TC-RULE-xxx`
- `TC-UI-xxx`
- `TC-EDGE-xxx`
- `TC-PERF-xxx`
- `TC-NFR-xxx`

### 12.2 例

- `TC-RULE-001`: 1ライン消去時の得点加算確認
- `TC-RULE-015`: 出現不能時にゲームオーバーへ遷移する
- `TC-UI-003`: 一時停止中に再開入力が有効である
- `TC-EDGE-008`: 左壁密着時に回転失敗し位置不変
- `TC-NFR-004`: 同一入力列で同一結果が再現される

### 12.3 テストケース記録項目

各テストケースには、少なくとも以下を持たせることを推奨する。

- TC ID
- タイトル
- 対応 FR / NFR / EXT / State
- 前提条件
- 入力 / 操作
- 期待結果
- 実施結果
- 備考

## 14. 優先順位

本プロジェクトでは、すべてを同一優先度で試験しない。  
優先順位は以下の通りとする。

### 14.1 最優先

- コアゲームルール
- 回転 / 衝突
- ゲームオーバー
- 一時停止
- 状態遷移
- 出現不能
- ライン消去と得点

### 14.2 高優先

- 画面表示整合性
- 設定画面挙動
- 数値表示整合性
- 再試行導線
- 無効入力の扱い

### 14.3 中優先

- 長時間プレイ時の基本安定性
- 素材差し替え時の表示成立性
- 予約状態の将来拡張確認

### 14.4 低優先 / 後続

- B-TYPE 詳細試験
- 保存機能詳細試験
- 装飾演出品質試験

- 見た目確認

## 16. 品質ゲート

以下を、本プロジェクトにおける基本的な品質ゲートとする。

### 16.1 文書ゲート

- コア10文書が揃っていること
- 主要 FR / NFR に対応先が存在すること
- 仕様と設計の大きな矛盾がないこと

### 16.2 ルールゲート

- ピース出現から固定までのサイクルが成立すること
- ライン消去が正しく処理されること
- 得点とレベル進行が仕様に沿うこと
- ゲームオーバー条件が成立すること

### 16.3 状態ゲート

- 一時停止中に進行しないこと
- ゲームオーバー後に通常プレイが継続しないこと
- タイトル、設定、プレイ、結果画面の導線が成立すること

### 16.4 境界ゲート

- 壁際回転
- 床際回転
- 出現不能
- 複数ライン同時消去
- 無効入力

### 16.5 非機能ゲート

- 重大な応答遅延がないこと
- 同一条件で結果が不安定にぶれないこと
- 文書からテストまで追跡可能であること

## 17. 変更時のテスト更新原則

仕様変更時は、コード修正のみで完了扱いにしてはならない。  
少なくとも以下を確認する。

- どの BR / UC / DM / SR / NSR に影響するか
- どの EXT に影響するか
- どの API 契約に影響するか
- 既存 TC のどれを更新すべきか
- 新規 TC が必要か
- 変更理由をどこへ記録するか

### 17.1 例: 回転仕様変更

たとえば wall kick を導入する場合、少なくとも以下を見直す必要がある。

- `24_piece_rotation_collision_spec.md`
- `32_state_machine_design.md`
- `43_test_cases_edge_conditions.md`
- 必要に応じて `14_non_functional_requirements.md`
- `16_traceability_matrix.md`
- `60_decision_log.md` または ADR

### 17.2 例: レベル進行変更

たとえばレベル上昇条件を変更する場合、少なくとも以下を見直す必要がある。

- `20_game_rules_spec.md`
- `41_test_cases_game_rules.md`
- `16_traceability_matrix.md`
- `61_change_log.md`

## 18. 完了判定との関係

本書はテスト方針文書であり、最終的な完了判定は `45_definition_of_done.md` で定義する。  
ただし、本書時点での基本認識として、以下を満たさない限り完了とはみなさない。

- 主要 FR に対応する TC が存在すること
- 主要 NFR に対する確認方針が存在すること
- コア画面遷移が確認されていること
- 回転 / 衝突 / 固定 / 消去 / ゲームオーバーが確認されていること
- 文書と実装とテストの関係が追えること

## 20. 結論

本プロジェクトにおけるテストは、実装の正しさ確認であると同時に、DocDD によって整備された文書群を下流成果物へ結び付けるための追跡装置である。

そのため、本書では次を最重要とする。

- BR を起点に 7 つの主要軸を追えること
- テストケースが FR / NFR / EXT / State と結び付いていること
- 変更時に影響箇所を辿れること
- 小規模題材でも実務的な試験方針が成立していること

本書を基準として、後続のテストケース文書では、ルール、UI、境界条件、性能確認を具体的に展開する。
