# 開発方針 / Development Policy

- 文書ID: DOC-MGT-050
- 最終更新日: 2026-03-23

## 1. 本書の目的
本書は、DocDD 見本プロジェクトを運用する際の基本判断原則を、日本語で簡潔に共有するための文書である。

## 2. 基本原則
1. 文書を先に直す
2. A-TYPE を主軸に判断する
3. 参照元との差異は必ず明記する
4. T-Spin のような独自拡張は upstream から downstream へ同時反映する
5. 言語選定は実装量より見通しを優先する

## 3. レイヤ分離原則
- 要求には「システムが満たすべき機能・品質」を書く
- 仕様には「外部から見える挙動」を書く
- 設計には「内部構造と責務分担」を書く
- テストには「何をどう検証するか」を書く
- B-TYPE 参考仕様保持や言語選定理由のような文書運用・プロジェクト運用事項は、要求へ混ぜず本書や `60_decision_log.md` で管理する

## 4. 変更同期原則
- 振る舞い変更時は、関連する要求・仕様・設計・試験文書を同一変更範囲で見直す
- 変更理由は `60_decision_log.md` または関連記録へ残す
- 非採用機能や制約も、仕様と試験から追跡可能に保つ
- 迷った場合は「文書を追加する」側へ倒す

## 5. レビュー観点
- 変更は A-TYPE 主軸と矛盾していないか
- 参照元との差異と採用理由が明示されているか
- 要求 / 仕様 / 設計 / 試験が混線していないか
- 上流判断が下流文書まで同期しているか
- 初学者が読んでも意図を誤解しにくいか
