## 1. 本書の目的

本書は、**「どこまでが Game Boy 版基準で、どこからが本見本の独自整理か」**を単独で読めるようにするための文書である。

DocDD の解説資料として、参照元に忠実な部分と、説明上の都合で追加・変更した部分を明確に分離する。

---

## 2. 整理方針

1. Game Boy 版を唯一の忠実再現対象とはしない
2. ただし操作・画面簡潔性・ゲームモード整理は主要参照元として尊重する
3. 実装対象は PC 単体アプリケーションとする
4. DocDD の説明価値が高い差分は意図的に追加する

## 4. 継承した基準

### 4.1 操作体系
- 十字キー、A、B、SELECT、START を論理入力として保持する
- PC 実装ではキーボードへ対応付けるが、文書では論理入力名を主表記とする

### 4.2 モード整理
- A-TYPE を主軸に据える
- B-TYPE は `25 lines` を含む参考仕様として残す

### 4.3 盤面と情報量
- 10×18 盤面
- NEXT 1 個
- シンプルな画面構成
- 余計な補助 UI を増やさない

---

## 5. 意図的に外した要素

### 5.1 実装対象から外したもの
- 対戦モード
- 通信要素
- モダンテトリス系の Hold
- Hard drop
- Ghost 表示
- Combo / Back-to-Back / Perfect Clear ボーナス

### 5.2 外した理由
- 文書体系の説明に対してスコープが広がりすぎるため
- 外部仕様・内部設計・テスト派生の説明が複雑になりすぎるため
- 初期見本では A-TYPE の縦方向の追跡性を優先するため

---

## 6. 独自拡張の理由

### 6.1 T-Spin を追加した理由
T-Spin は Game Boy 版準拠の忠実再現ではないが、以下の理由で採用する。

- 既存参照元からの変更要求を表現しやすい
- ルール仕様、得点仕様、回転仕様、設計、試験への波及が明確である
- DocDD の変更管理見本として教育効果が高い

### 6.2 PC キー写像を追加した理由
- 実行環境が PC であるため、論理ボタンと物理入力の対応が必要
- UI 仕様、入力仕様、内部契約の境界を説明しやすい

## 7. この文書の使い方

- GitHub 初見者には「まず何を参照元にし、何を変えているのか」を示す入口として使う
- 仕様レビュー時には、差分がスコープ逸脱か意図的設計かを確認するために使う
- 変更要求レビュー時には、継承項目・独自項目のどちらに影響するかを確認するために使う
