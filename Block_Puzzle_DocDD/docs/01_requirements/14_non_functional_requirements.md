## 1. 本書の目的

本書は、落下ブロックゲームを題材とした DocDD 見本プロジェクトにおいて、**システムと文書群が満たすべき品質要求** を定義する文書である。

本書が扱う品質は、操作応答性、ルール再現性、可読性、保守性、異常系耐性、テスト容易性、文書整合性である。

---

## 2. 非機能要求の定義方針

1. DocDD 見本として重要な品質観点を優先する
2. Game Boy 版基準の挙動を再現しやすい粒度で記述する
3. 実装言語やライブラリに依存しすぎない表現で記述する
4. 設計・テストで確認可能な要求とする
5. 実装物だけでなく文書群にも適用する

---

## 3. 要求 ID 体系

- NFR-001〜099: 操作性・応答性
- NFR-100〜199: 再現性・正確性
- NFR-200〜299: 可読性・保守性
- NFR-300〜399: 安定性・異常系耐性
- NFR-400〜499: テスト容易性・検証性
- NFR-500〜599: 文書整合性・トレーサビリティ
- NFR-600〜699: 拡張性・移植性

## 4. 非機能要求一覧

### 4.1 操作性・応答性

#### NFR-001 基本操作応答性
システムは、通常状態ではプレイヤー入力に対する結果を次の更新単位以内に反映できなければならない。

#### NFR-002 入力解釈の一貫性
システムは、同一状態・同一条件下で同種入力に対し一貫した挙動を示さなければならない。

#### NFR-003 状態依存入力制御の明確性
システムは、状態ごとに有効入力と無効入力の扱いが明確でなければならない。

#### NFR-004 表示更新の非阻害性
システムは、表示更新がゲーム進行や入力応答を不自然に阻害しない構造でなければならない。

#### NFR-005 NEXT 表示切替の可視性
システムは、SELECT による NEXT 表示 ON/OFF がプレイヤーに誤解なく認識される UI でなければならない。

#### NFR-006 表示基準解像度の整合
システムは、想定描画解像度 640×480 において、盤面・NEXT・SCORE・LINES・LEVEL を視認可能に配置し、PC 利用前提の固定基準解像度として一貫運用できること。

### 4.2 再現性・正確性

#### NFR-101 ルール処理の一貫性
システムは、接地、固定、ライン消去、得点加算、進行更新等のルール処理を同一条件下で一貫して行わなければならない。

#### NFR-102 判定結果の明確性
システムは、盤面外判定、衝突判定、ライン完成判定、ゲームオーバー判定等について曖昧な解釈を生じにくい構造でなければならない。

#### NFR-103 プレイ結果の再現可能性
システムは、同一初期条件と同一入力列に対して同一のプレイ結果を再現しやすい構造でなければならない。

#### NFR-104 時間進行の安定性
システムは、固定更新幅または等価の手段により、落下速度表、ARE、pause 再開、入力優先順の解釈を崩さない時間管理を持たなければならない。

#### NFR-105 表示情報の整合性
システムは、盤面状態、スコア、ライン数、レベル、NEXT 表示状態が内部状態と矛盾しないようにしなければならない。

#### NFR-106 ベースライン準拠性
システムは、Game Boy 基準として採用した速度表、スコア倍率、A/B 回転、NEXT 表示仕様から無意識に逸脱しないよう設計・レビューできなければならない。

### 4.3 可読性・保守性

#### NFR-201 コードの見通しの良さ
システムは、個人または少人数で保守可能な範囲でコードの見通しが良い構造でなければならない。

#### NFR-202 責務分離
システムは、入力処理、ゲーム進行、ルール判定、画面表示、状態遷移の責務が過度に混在しない構造でなければならない。

#### NFR-203 変更影響の追跡容易性
システムは、主要仕様を変更した場合に、どの文書・設計・実装・テストへ影響するか追いやすい構造でなければならない。

#### NFR-204 局所修正容易性
システムは、小規模な仕様変更やバグ修正を必要以上に広範囲へ波及させずに行いやすい構造でなければならない。

#### NFR-205 命名の一貫性
システムおよび文書は、主要概念に対して一貫した名称を用いなければならない。

#### NFR-206 過剰技術化の抑制
システムは、本プロジェクトの規模と目的に対して過剰な技術要素を導入しないことを原則としなければならない。

### 4.4 安定性・異常系耐性

#### NFR-301 想定内異常に対する破綻回避
システムは、不正な状態遷移要求や想定外入力タイミングに対して可能な限り致命的破綻を避けなければならない。

#### NFR-302 不正状態の早期検出容易性
システムは、不正状態や矛盾状態が発生した場合に、それを検出または切り分けしやすい構造でなければならない。

#### NFR-303 非採用入力の無害性
システムは、Hold や Hard drop のような非採用入力が誤って有効化されてもプレイ進行を破壊しない構造でなければならない。

#### NFR-304 表示切替の安全性
システムは、NEXT 表示 ON/OFF 切替時に内部キューや進行順を破壊してはならない。

### 4.5 テスト容易性・検証性

#### NFR-401 ルール単体試験容易性
システムは、落下、固定、得点、レベル進行、T-Spin 判定などをヘッドレスに検証しやすい構造でなければならない。

#### NFR-402 状態遷移試験容易性
システムは、タイトル、設定、プレイ、一時停止、ゲームオーバーを状態遷移単位で検証しやすい構造でなければならない。

#### NFR-403 図面同期検証性
状態遷移図やフローチャートは、本文との整合性をレビューで検証しやすい粒度で管理されなければならない。

#### NFR-404 受入観点の検証可能性
各コア文書の受入観点は、レビュー時に Yes/No 判定できる粒度でなければならない。

### 4.6 文書整合性・トレーサビリティ

#### NFR-501 文書役割分離
要求、仕様、設計、テスト、記録の役割が文書間で混在しないこと。

#### NFR-502 上下流追跡可能性
少なくとも主要機能について、BR→UC→DM→SR/NSR→EXT→Design→TC を辿れること。

#### NFR-503 基準と差分の可視性
Game Boy 準拠要素と独自拡張要素が 1 文書以上で明確に比較できること。

#### NFR-504 変更記録完全性
仕様差分やレビュー指摘は、Decision Log / Change Log / Review Log のいずれかへ必ず残されなければならない。

---

## 5. Acceptance-Driven 観点

1. 応答性、再現性、追跡性、図同期が非機能要求として定義されていること
2. Game Boy 基準からの無意識な逸脱を防ぐ要求が含まれていること
3. SELECT/NEXT 切替の安全性・可視性が品質要求へ入っていること
4. 文書そのものの品質要求が定義されていること
5. テスト設計へ接続できる粒度の NFR になっていること
