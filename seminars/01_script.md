## スライド7：現場向けツールで現場がコケるパターン

「動いた」の先にある、運用破綻の壁を超える。  
AI が実装を担う際には、人間がこれらの要素を定義すると価値が高まる

- 🗄️ **技術スキーマの欠如**  
  形や制約が曖昧で、汚いデータが入った瞬間にシステムが壊れる。

- 🔄 **非機能要件の無視**  
  再実行性やログがなく、トラブル時に「何がおきたか」追跡できない。

- 👤 **運用設計の不在**  
  誰がエラーを直し、誰が取込を承認するか決まっておらず放置される。

- 🔧 **保守性の低さ**  
  入力形式の変更に弱く、追加や少しの変更で大きな改修が必要になる。

- 🛡️ **ガバナンスの欠如**  
  権限管理や監査ログがなく、企業として説明責任を果たせない。

## スライド8：要求をそのまま実装へ流し込まない

最初に決めるべきは実装方法ではなく、**問題の性質**である。

**事例：現場からの要求**  
「DX トレンドを受けて、Excel で管理している実績データを見える化しろと指示がきた」  
→ そのままでは要件になっていない

**まず確認すべきこと「業務モデル」と「評価 KPI」**

1. どこに、どんなデータがあるか（種類）
2. どんな頻度で更新されるか（時間）
3. 記録単位とデータ量は（件数・期間）
4. 何をどの粒度で管理・分析したいのか（成果物）

**ここで初めて判断する**  
1. Excel で足りるか　2. DB 化が必要か　3. Web 画面が必要か

**方式を決めた後に詰めること**  
1. 列と型　2. 同一判定キー　3. 例外の扱い　4. 誰がいつ取り込むか　5. 完成判定

## スライド9：最初に定義するのは「業務モデル」と「評価 KPI」

前ページで見た最初の論点を具体化する。**まず定義すること**

**1. 業務モデル**
- 記録単位は何か・工程はどう流れるか・何を 1 レコードとみなすか

**2. 評価 KPI**
- 最終的に何を見たいのか・何を判断したいのか・どの粒度で集計したいのか

**今回の題材で定義しているもの**
- 記録単位：1 製番 ＝ 1 台、各工程完了ごとに 1 レコード
- 工程：内装組立 → 外装組立 → 出荷検査
- KPI：工程別時間別の作業実績数、作業者ごとの日別作業台数、工程間の滞留数

## スライド18：DocDD とは？

**DocDD = Document-Driven Development**

プロンプト経由の AI コーディング（バイブコーディング）に対して、コード製作の前に詳細な仕様書や設計書（ドキュメント）を、一次情報として作成し、それを「契約」として実装・テストを進める手法。  
AI でドキュメントからコードを生成し、仕様との整合性を保つ。手戻りの削減や、AI 時代における仕様の明確化（API ファースト等）に強みがある。

- 文書を増やすことが目的ではない
- 実装判断を先に文書で固定する
- 要求 / 仕様 / 設計 / 試験 / 変更記録をコードと文章で繋ぐ
- 変更時に「どこを直すか」を追える
- 大規模・全体整合・複数人開発で効く

> 仕様書や設計書作成も AI に支援してもらいながら進めるのも良

## スライド22：デモ方針：3分クッキング

AI にコードを書かせる様子を一緒に確認しますが、**実行は事前検証済み版で行います。**

| 機能 | 内容 |
|---|---|
| **重複が増えない** | DB の UNIQUE 制約で防衛できる。同じファイルを 2 回投入しても件数が変わらない。複数 PC から同時に登録しても安全。 |
| **例外は除外される** | 入力不正レコードは DB に入れない。reject テーブルに理由付きで残る。現場で修正 / 再投入も可能。 |
| **集計出力** | 期間、設備、工程でフィルタ。CSV 出力で会議に持ち出せる。見える化で終わらせない。 |

> 論点は「生成できるか」ではなく「運用に耐えるか（仕様＝3文書）」にあるためです。

## スライド23：利用する技術スタック

### 実装構成

| 役割 | 技術 | 詳細 |
|---|---|---|
| 取込処理 | **Python** | 列の読み替え、型変換、各種チェック、正常 / reject 振り分け |
| 保存先 | **PostgreSQL** | 正常データ、reject、取込履歴を保存 |
| 画面 | **Streamlit** | アップロード、検証、プレビュー、登録 |

### 処理フロー

```
CLI 入力  ─┐
           ├─→  共通ロジック（Python）  ─→  データベース（PostgreSQL）  ─→  Web 表示（Streamlit）
Web 入力  ─┘
```

**共通化の重要性**  
入口が違っても、判定・変換・登録のロジックは共通化する。  
将来の仕様変更や、大きなシステムへの組込みにも強い。
