## スライド7：現場向けツールで現場がコケるパターン

「動いた」の先にある、運用破綻の壁を超える。  
AIが実装を担う際には、人間がこれらの要素を定義すると価値が高まります。

- **技術スキーマの欠如**  
  形や制約が曖昧で、汚いデータが入った瞬間に壊れる。

- **非機能要件の無視**  
  再実行性やログがなく、トラブル時に何が起きたか追跡できない。

- **運用設計の不在**  
  誰がエラーを直し、誰が取込を承認するか決まっておらず、止まりやすい。

- **保守性の低さ**  
  入力形式の変更に弱く、少しの変更で大きな改修が必要になる。

- **ガバナンスの欠如**  
  権限管理や監査ログがなく、企業として説明責任を果たせない。

## スライド9：最初に定義するのは「業務モデル」と「評価KPI」


## スライド18：DocDDとは？

**DocDD = Document-Driven Development**

プロンプト経由のAIコーディングに対して、コード作成の前に詳細な仕様書や設計書を一次情報として作成し、それを「契約」として実装・テストを進める手法です。  
AIでドキュメントからコードを生成し、仕様との整合性を保ちます。手戻りの削減や、AI時代における仕様の明確化に強みがあります。

- 文書を増やすことが目的ではない
- 実装判断を先に文書で固定する
- 要求 / 仕様 / 設計 / 試験 / 変更記録を、コードと文章でつなぐ
- 変更時に「どこを直すか」を追える
- 大規模・全体整合・複数人開発で効く

> 仕様書や設計書の作成も、AIに支援してもらいながら進めるのが良い。

## スライド22：デモ方針：3分クッキング

AIにコードを書かせる様子は見せますが、**実行は事前検証済み版で行います。**

| 観点 | 内容 |
|---|---|
| 重複が増えない | DBのUNIQUE制約で防衛できる。同じファイルを2回投入しても件数が変わらない。複数PCから同時に登録しても安全。 |
| 例外は除外される | 入力不正レコードはDBに入れない。rejectテーブルに理由付きで残る。現場で修正 / 再投入も可能。 |
| 集計・出力 | 期間、設備、工程でフィルタ。CSV出力で会議に持ち出せる。見える化で終わらせない。 |

> 論点は「生成できるか」ではなく、**「運用に耐えるか（仕様＝3文書）」** にあります。

## スライド23：利用する技術スタックの解説

### 技術スタック

| 役割 | 技術 | 詳細 |
|---|---|---|
| 取込処理 | Python | 列の読み替え / 型変換 / 必須チェック / 値域チェック / 正常・reject振り分け |
| 保存先 | PostgreSQL | 正規化データ / reject / 取込履歴を保存 |
| 画面 | Streamlit | アップロード → 検証 → プレビュー → 登録 |

### 処理フロー

```text
CLI入力 ─┐
         ├→ 共通ロジック処理（Python） → データベース（PostgreSQL）
Web入力 ─┘                                  └→ Web表示（Streamlit）
```

### 共通化の重要性
入口が違っても、判定・変換・登録のロジックは共通化する。  
将来の仕様変更や、大きなシステムへの組込みにも強い。
