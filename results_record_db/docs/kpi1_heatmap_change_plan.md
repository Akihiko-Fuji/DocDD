# KPI1 工程別・時間帯別ヒートマップ 改修計画

配置パス: `results_record_db/docs/kpi1_heatmap_change_plan.md`

## 1. 文書の位置づけ

本書は、`results_record_db` のKPI1画面へヒートマップを追加するための改修仕様書である。
`results_record_db` はDocDDセミナーの為に用意されたものであり、
本ドキュメント`kpi1_heatmap_change_plan.md`はセミナー中のcodexによるコード改修のデモ実演を目的に使用する。

- 既存機能の仕様は `results_record_db/README.md` の「仕様」欄を正とする。
- 今回追加する表示仕様は、本書の「仕様」および「受入条件」を正とする。
- 本書の「解説」と「実装上の補足」は非規範とする。
- 両文書の仕様が矛盾する場合は実装を中断し、矛盾箇所を報告する。推測による補完は行わない。

## 2. 目的

### 仕様

KPI1タブに、既存の工程別・時間帯別棒グラフを残したまま、同じ集計結果を使用したヒートマップを追加する。

ヒートマップにより、工程と時間帯の組合せごとの完了台数を、数値と色の濃さの両方で確認できるようにする。

### 解説

棒グラフは工程間の件数比較に向く一方、ヒートマップは実績が集中する時間帯や工程ごとの偏りを一覧で把握しやすい。

今回の改修ではKPIの定義や集計方法を変更しない。既存データの別の見せ方を、仕様として追加する。

## 3. 対象

### 仕様

主な変更対象は次のファイルとする。

- `results_record_db/src/streamlit_app.py`
- `results_record_db/tests/test_kpi.py`

新しい外部ライブラリは追加しない。既存のAltair、Pandas、Streamlitを使用する。

## 4. KPI1集計仕様

### 仕様

ヒートマップは、既存の `build_kpi1()` が返すDataFrameをそのまま使用する。

集計条件は次のとおりとする。

| 項目 | 仕様 |
|---|---|
| 集計対象 | 画面で選択された期間・工程・作業者に一致する `work_log` |
| 完了時刻 | `end_ts` |
| 時間帯 | `end_ts` の時を1時間単位で集計 |
| 表示時間 | 08:00～17:00 |
| 横軸 | `hour_slot` |
| 縦軸 | `process_name` |
| セルの値 | `work_count` |
| 日付の扱い | 選択期間内の全日を、同じ時間帯・工程ごとに合算 |
| 工程順 | 内装組立 → 外装組立 → 出荷検査 |
| 未実績セル | 0として表示 |

既存の期間・工程・作業者フィルタをヒートマップにも適用する。

`build_kpi1()` の集計定義、返却列および既存棒グラフの集計結果は変更しない。

### 解説

例えば期間に複数日を指定した場合、08:00のセルは選択期間内に08:00～08:59で完了した実績の合計を示す。日別の内訳は今回のヒートマップでは表示しない。

## 5. ヒートマップ表示仕様

### 仕様

ヒートマップは、KPI1タブ内の既存棒グラフの直後、CSVダウンロードボタンおよびデータ表の前に表示する。

表示内容は次のとおりとする。

| 表示要素 | 仕様 |
|---|---|
| グラフタイトル | `工程別・時間帯別の完了台数ヒートマップ` |
| 横軸タイトル | `完了時間帯` |
| 横軸順 | 08:00から17:00まで昇順 |
| 縦軸タイトル | `工程` |
| 縦軸順 | 内装組立 → 外装組立 → 出荷検査 |
| 色 | `work_count` が多いほど濃くする |
| 色スケール | 最小値0、色範囲 `#f7fbff` → `#6baed6` の連続色 |
| 凡例タイトル | `完了台数` |
| セル内表示 | `work_count` を整数で表示 |
| セル内文字色 | `#111827` |
| ツールチップ | 時間帯、工程、完了台数 |

色スケールはAltairの量的尺度を使用する。0件を `#f7fbff` とし、選択条件内の最大値を `#6baed6` とする。

セル内の数値は `#111827` で全セルに表示する。KPI1集計結果が空でない場合、0件のセルも空白にせず `0` と表示する。

画面で工程を一部だけ選択した場合は、選択された工程だけを固定工程順に従って表示する。

対象データがない場合は、既存動作と同様にグラフを表示せず、例外を発生させない。

### 表示イメージ

次の数値は表示例であり、テストの期待値ではない。

| 工程＼時間 | 08:00 | 09:00 | 10:00 | 11:00 | 12:00 | 13:00 | 14:00 |
|---|---:|---:|---:|---:|---:|---:|---:|
| 内装組立 | 42 | 51 | 48 | 53 | 2 | 46 | 49 |
| 外装組立 | 35 | 44 | 56 | 41 | 1 | 52 | 47 |
| 出荷検査 | 8 | 15 | 19 | 23 | 0 | 41 | 52 |

実画面では15:00、16:00、17:00も表示する。

## 6. 既存機能の維持

### 仕様

次の既存機能は変更しない。

- KPI1の棒グラフ
- KPI1のCSVダウンロード
- KPI1の集計結果表
- 期間・工程・作業者フィルタ
- Importタブ
- KPI2工程間滞留
- KPI3作業者別日別実績
- DB接続、データ取込、正規化およびreject処理

棒グラフ、ヒートマップ、CSV、集計結果表は同じ `build_kpi1()` の結果を使用する。

## 7. 対象外

### 仕様

今回の改修では次を行わない。

- KPI1の計算式または集計粒度の変更
- 棒グラフからヒートマップへの置換
- 日付を軸に追加した日別ヒートマップ
- 目標値、警告値、良否判定または閾値の追加
- 工程能力やボトルネックの自動判定
- DBスキーマまたはSQLの変更
- ファイル構成の再編
- KPI1以外の画面デザイン変更
- 新しい外部ライブラリの追加
- 今回の表示追加と無関係なリファクタリング

## 8. 受入条件

以下をすべて満たした場合に改修完了とする。

1. KPI1タブに既存棒グラフが残っている。
2. 既存棒グラフの直後にヒートマップが表示される。
3. ヒートマップの横軸に08:00～17:00が昇順で表示される。
4. ヒートマップの縦軸に、選択された工程だけが固定工程順で表示される。
5. 各セルに整数の完了台数が表示される。
6. KPI1集計結果が空でない場合、実績がない工程・時間帯のセルに `0` が表示される。
7. 完了台数が多いセルほど色が濃く表示される。
8. セルのツールチップで時間帯、工程、完了台数を確認できる。
9. 期間・工程・作業者フィルタの変更が、棒グラフとヒートマップの両方へ同じように反映される。
10. ヒートマップのセル値が、同じ条件のKPI1集計結果表と一致する。
11. KPI1のCSV列、ファイル名およびダウンロード機能が変わっていない。
12. Import、KPI2、KPI3の動作が変わっていない。
13. 対象データがない場合に例外が発生しない。
14. 新しい依存ライブラリが追加されていない。
15. `python -m pytest -q` が成功する。

## 9. テスト仕様

### 自動テスト

既存テストを維持し、少なくとも次を確認するテストを `results_record_db/tests/test_kpi.py` に追加する。

- 3工程を選択した場合、08:00～17:00の10時間 × 3工程 = 30セル分の集計行が生成される。
- 実績が存在しない工程・時間帯の `work_count` が0になる。
- 工程を一部だけ選択した場合、選択していない工程の集計行が含まれない。
- 既存の期間・工程・作業者フィルタによる集計結果が維持される。

### 手動確認

1. `results_record_db` へ移動する。
2. `python -m pytest -q` を実行する。
3. `python -m streamlit run src/streamlit_app.py` で画面を起動する。
4. KPI1タブを開く。
5. 3工程選択時に3行、08:00～17:00の10列が表示されることを確認する。
6. 工程を1つに絞り、ヒートマップも1行になることを確認する。
7. 作業者を変更し、棒グラフ、ヒートマップ、集計結果表の件数が連動することを確認する。
8. 0件のセルが空白ではなく `0` と表示されることを確認する。

## 10. 実装上の補足

本節は非規範とする。

- 既存の `kpi1` DataFrameを、Altairの `mark_rect()` と `mark_text()` で重ねると実装しやすい。
- 横軸は `hour_order` を使って並び順を固定する。
- 縦軸は `PROCESS_FLOW` を使って工程順を固定する。
- ヒートマップ専用の再集計は行わず、棒グラフと同じDataFrameを再利用する。
- 変更差分を小さく保つため、KPI1タブの描画処理以外は変更しない。
